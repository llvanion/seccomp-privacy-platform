# Server-Client API Documentation

> Current status (2026-06-01): this WebSocket API is a legacy/demo interface.
> The network wire format was migrated from Python `pickle` to a JSON/base64
> envelope in `frontend.common.wire`. The interface is still local/demo by
> default because it does not provide production authentication or transport
> hardening. The server refuses non-loopback binds unless
> `SSE_ALLOW_LEGACY_PICKLE_WS=1` is set as an explicit demo-only override.
> `SSE_PRODUCTION_MODE=1` retires this interface for production and refuses
> startup even on loopback and even with the demo override enabled. Production
> jobs should use the query workflow / bridge pipeline APIs, not this legacy
> WebSocket.

## 1. Overview
The communication between the client and server relies on a long-lived **WebSocket** connection. WebSocket frames use UTF-8 JSON generated by `frontend.common.wire.dumps_message()` and parsed by `frontend.common.wire.loads_message()`. Opaque bytes from SSE schemes are represented as base64 markers inside JSON; Python `pickle` is not accepted on the network path.

## 2. General Message Envelope
Each WebSocket frame is a JSON envelope:

```json
{
  "schema": "sse.frontend.ws/v1",
  "payload": {
    "type": "<MsgType>",
    "sid": "<service_id>",
    "content": {"__sse_wire_type__": "bytes", "base64": "<base64>"},
    "request_id": "<optional uuid>",
    "token_digest": {"__sse_wire_type__": "bytes", "base64": "<base64>"}
  }
}
```

After decoding, client and server code continue to work with the following Python dictionary shape:

```python
{
    "type": "<MsgType>",       # required: The operation or instruction type
    "sid": "<service_id>",     # required: The unique identifier for an SSE service instance
    "content": b"<...>",       # optional: raw scheme bytes or structured JSON content bytes
    # ...other metadata like token_digest, request_id based on the instruction type
}
```

Structured `content` dictionaries are encoded with `frontend.common.wire.encode_content()` and decoded with `decode_content()`. These bytes are UTF-8 JSON and may contain nested bytes via the same base64 marker.

The server manages a state machine for each `sid`:
- `NOT_EXISTS (0)`
- `CONFIG_UPLOADED_BUT_EDB_NOT_UPLOADED (1)` 
- `ALL_READY (2)`

## 3. Operations / API Endpoints

### 3.1 INIT (Initialization Handshake)
- **Purpose**: Sent when establishing the WebSocket connection to activate or initialize the service session on the server.
- **Request**:
  - `type`: `init`
  - `sid`: `<service_id>`
  - `content`: (None)
- **Response**:
  - `type`: `init`
  - `content` (structured JSON bytes): `{"ok": True, "state": <ServiceState Integer>}`

### 3.2 CONFIG (Upload Configuration)
- **Purpose**: Uploads SSE configuration details to the server. Must be preceded by `INIT`.
- **Request**:
  - `type`: `config`
  - `content` (structured JSON bytes): Python dictionary containing configuration data.
- **Response**:
  - `type`: `config`
  - `content` (structured JSON bytes): `{"ok": True}`

### 3.3 UPLOAD_DB (Upload Encrypted Database)
- **Purpose**: Uploads the client-side encrypted database (EDB) to the server for persistence.
- **Request**:
  - `type`: `upload_edb`
  - `content`: native `bytes` from serializing the EDB itself, base64-encoded by the outer JSON envelope.
- **Response**:
  - `type`: `upload_edb`
  - `content` (structured JSON bytes): `{"ok": True}` or `{"ok": False, "reason": "..."}`

### 3.4 TOKEN (Single Search Query)
- **Purpose**: Issues a single search query using an SSE Search Token.
- **Request**:
  - `type`: `token`
  - `content`: Native `bytes` array representing the serialized `SSEToken`.
  - Additional field: `token_digest` as bytes (`sha256(token_bytes).digest()`).
- **Response**:
  - `type`: `result`
  - `content`: Native `bytes` containing the serialized search results or structured JSON bytes `{"ok": False, "reason": "..."}` on error.
  - Additional field: Returns `token_digest` as received.

### 3.5 MULTI_TOKEN (Batch Search Query)
- **Purpose**: Bulk dispatch multiple Search Tokens to reduce network overhead in multi-keyword searches.
- **Request**:
  - `type`: `multi_token`
  - `content` (structured JSON bytes): `{"tokens": [{"token_bytes": b"...", "token_digest": b"..."}, ...]}`
  - Additional field: `request_id` (UUID string)
- **Response**:
  - `type`: `multi_result`
  - `content` (structured JSON bytes): `{ "ok": True, "results": [ {"token_digest": b"...", "result": b"<...>"}, ... ] }`
  - Additional field: Returns `request_id` as received.

### 3.6 DELETE (Delete Data)
- **Purpose**: Removes specific entries in the encrypted DB.
- **Request**:
  - `type`: `delete`
  - `content` (structured JSON bytes): `{ "token_bytes": b"<...>" (optional), "indices": [<int>, ...] (optional) }`
  - Additional field: `request_id` (UUID string)
- **Response**:
  - `type`: `delete_result`
  - `content` (structured JSON bytes): `{"ok": True, "deleted_count": <int>}`
  - Additional field: Returns `request_id` as received.

### 3.7 UPDATE (Update / Insert Data)
- **Purpose**: Adds or modifies ciphertexts in the encrypted database.
- **Request**:
  - `type`: `update`
  - `content` (structured JSON bytes):
    ```python
    {
       "token_bytes": b"<...>",   # (optional)
       "encrypted_data": bytes | list[bytes] | None,  # Ciphertext list or items
       "entries": [{"addr": str | bytes, "value": bytes}, ...] # (optional) Direct address bindings
    }
    ```
  - Additional field: `request_id` (UUID string)
- **Response**:
  - `type`: `update_result`
  - `content` (structured JSON bytes): `{"ok": True, "updated_count": <int>}`
  - Additional field: Returns `request_id` as received.

### 3.8 CONTROL (Server Control Event)
- **Purpose**: Unsolicited server-side push events (e.g. for connection eviction, warnings).
- **Request**: None (Server initiated)
- **Response**:
  - `type`: `control`
  - `content`: UTF-8 encoded `bytes` containing the message string.
