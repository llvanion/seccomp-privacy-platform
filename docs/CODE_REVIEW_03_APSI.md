# Code Review — Step 3: A-PSI / PJC Module

**Scope:** `a-psi/moduleA_psi/scripts/`, `a-psi/moduleA_psi/scripts/validate_bridge_job.py`, `a-psi/moduleA_psi/scripts/policy_release.py`

---

## 1. Module Purpose

The A-PSI module is the PJC execution and result-governance layer. It:

1. Validates bridge-generated job metadata before running PJC.
2. Runs the Private Join and Compute (PJC) binary over `server.csv` / `client.csv`.
3. Applies policy-controlled release: threshold check, rate limit, optional authentication, duplicate-query denial, optional differential privacy noise, and public report generation.

The underlying PJC binary is from Google's `private-join-and-compute` project (Bazel-built). The module wraps it with governance and audit semantics.

---

## 2. `validate_bridge_job.py` — Pre-PJC Validation

This script runs before PJC and gates execution on the bridge metadata contract:

```python
KNOWN_NORMALIZER_SCHEMA_VERSIONS = {"normalizer-schema/v1"}
KNOWN_NORMALIZERS = {"identity", "email", "phone"}
```

Checks performed:
- `schema` is `bridge_job_meta/v1` (or absent for legacy jobs).
- `bridge` section exists and contains all required keys.
- `token_scope`, `token_scheme`, `token_key_version`, `normalize_version` are non-empty.
- `normalizer_schema_version` is present and in `KNOWN_NORMALIZER_SCHEMA_VERSIONS`.
- `bridge.server.normalizer` and `bridge.client.normalizer` are in `KNOWN_NORMALIZERS`.
- `server.csv` and `client.csv` exist and are non-empty.
- CSV row counts match `job_meta.json` `counts.*_unique_join_tokens` values.

This validation prevents jobs generated by unrecognized normalizer implementations from running PJC. The `KNOWN_NORMALIZER_SCHEMA_VERSIONS` set acts as a code-level allowlist that must be updated when a new normalizer algorithm is deployed.

## 3. `policy_release.py` — Result Governance

### 3.1 Overall Decision Flow

```
Load PJC result (attribution_result.json)
  → Optional: check auth (HMAC-SHA256, timestamp, nonce)
  → Check rate limit (count prior releases for same caller/window)
  → Optional: deny duplicate query (canonical query signature)
  → Threshold check (intersection_size ≥ k)
  → Optional: apply DP noise (Laplace mechanism)
  → Optional: round sum to nearest N
  → Write public_report.json
  → Append policy_audit/v1 record
```

### 3.2 Threshold Policy (`apply_threshold_policy`)

```python
if size < threshold_k:
    return {"decision": "deny", "reason_code": "below_k", ...}
```

Threshold is the primary privacy gate: if fewer than `k` entities appear in the intersection, the result is not released. This prevents a near-trivially small intersection from leaking membership.

### 3.3 Rate Limiting (`apply_rate_limit`)

The rate limit is enforced by scanning the `audit_log.jsonl` for prior `policy_release` events with the same `caller` and `window` (start/end timestamps). If `used >= max_queries`, the request is denied.

**Limitation:** This is a linear scan of the audit log file. For high-volume production use, this would need an indexed store. For the current prototype, it is acceptable.

### 3.4 Duplicate Query Denial (`seen_query_signature`)

A canonical query signature is computed as:
```
SHA-256(job_id | "|" | threshold_k | "|" | caller | "|" | window_start | "|" | window_end)
```

If the same canonical signature appears in the audit log for the same caller, the request is denied with `reason_code = duplicate_query`. This prevents an adversary from repeating the exact same query to average out noise or probe the system.

### 3.5 Authentication (`authenticate_request`)

Optional HMAC-SHA256 authentication:
- `key_id` identifies the signing key in `auth_config.json`.
- Message = `key_id \n caller \n timestamp \n nonce \n job_id \n query_sig` (newline-joined).
- HMAC is compared with `hmac.compare_digest` (constant-time).
- Timestamp must be within `max_clock_skew_sec` seconds of current time.
- Nonce must not have appeared before for the same `key_id` (replay protection via audit log scan).

### 3.6 Differential Privacy

Optional Laplace mechanism (`dp_epsilon`, `dp_sensitivity`):
```python
noise = -scale * sign(u) * log(1 - 2|u|)   # inverse-CDF Laplace
scale = sensitivity / epsilon
```

Noise is added to the released `intersection_sum`. This is an optional flag — not enforced by default.

**Note:** The implementation uses Python's `random.uniform` rather than a cryptographically secure RNG for the Laplace noise. For production DP this should use `secrets.SystemRandom` or a proper DP library.

### 3.7 Value Mode Decoration

When `value_mode = "amount"`, the released sum is interpreted as cents and decorated with:
- `intersection_sum_cents`: raw integer cents value.
- `intersection_sum_eur`: decimal string in EUR.
- `intersection_sum`: set to the EUR string for display.

This is the format used in the demo (`intersection_sum = 425` = €4.25).

### 3.8 Audit Record (`policy_audit/v1`)

Written to `audit_log.jsonl`:
```json
{
  "schema": "policy_audit/v1",
  "event": "policy_release",
  "job_id": "...",
  "correlation_id": "...",
  "caller": "...",
  "decision": "allow" | "deny",
  "reason_code": "...",
  "threshold_k": N,
  "rate_limit": {...},
  "pjc_result_sha256": "...",
  "canonical_query_signature": "...",
  "duration_ms": N
}
```

## 4. Run Scripts

| Script | Purpose |
|---|---|
| `run_pjc.sh` | Single-machine: runs PJC server and client on localhost simultaneously |
| `run_pjc_job_server.sh` | Two-machine: runs server side of a prepared job |
| `run_pjc_job_client.sh` | Two-machine: runs client side of a prepared job, merges buckets |
| `run_pjc_bucketed_server.sh` | Two-machine bucketed: iterates through bucket manifest on server side |
| `run_pjc_bucketed_client.sh` | Two-machine bucketed: iterates through bucket manifest on client side |

The single-machine `run_pjc.sh` is what the demo uses. It runs both `server` and `client` PJC processes on the same machine with loopback gRPC.

## 5. Identified Gaps / Observations

| Item | Severity | Note |
|---|---|---|
| Rate limit via audit log scan | Medium | O(log_size) at every release request; needs indexed store for production |
| Nonce replay check via audit log scan | Medium | Same issue — O(log_size) linear scan |
| Laplace noise uses `random.uniform` | Medium | Not cryptographically random; production DP should use `secrets.SystemRandom` or a vetted DP library |
| Phone normalizer equivalence not enforced cross-module | Low | Bridge and any other normalizers must agree on the same algorithm; no cross-check currently |
| `rate_limit.max` is per-window per-caller | Informational | Does not aggregate across callers or tenants; each caller has their own counter |
| `seen_query_signature` scans full log | Low | Works for demo; would need an indexed duplicate-query store for high volume |
| PJC binary must be pre-built | Informational | Requires Bazel build of `private-join-and-compute`; a `PJC_BIN_DIR` env override allows distributing pre-built binaries |

---

## 6. Summary

The A-PSI module provides a complete policy-release governance layer over PJC results. The bridge job validator correctly gates PJC execution on the normalizer contract. The policy release script implements all five required controls: threshold, rate limit, duplicate query denial, optional auth, and optional DP. The main engineering risks are the O(n) audit log scans (acceptable for prototype scale) and the non-cryptographic RNG for Laplace noise. The module correctly avoids publishing results below the threshold `k` and records every decision in an auditable JSONL log.
