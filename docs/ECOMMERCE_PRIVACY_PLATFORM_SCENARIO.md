# 电商隐私库平台场景说明

## 1. 平台定位

这个项目更适合被描述为一个：

1. 面向电商场景的高隐私数据平台
2. 结合 `SSE + 受控恢复 + token bridge + PJC + policy release` 的隐私查询与隐私协作平台

它不是一个“所有参与方都能直接查明文数据”的普通数据库。

它的目标是：

1. 平台自己持有大量高敏感电商数据
2. 平台内部也不能随意明文扩散这些数据
3. 外部合作方只能在最小必要范围内获得结果
4. 外部合作方之间可以协作，但不能互相直接交换底层用户明文

一句话理解：

1. `SSE` 负责在平台内部先把“需要动哪些记录”这件事尽量缩小
2. `PJC` 负责在跨机构协作时只暴露“交集上的允许结果”

## 2. 核心参与方

建议把平台里的角色分成下面几类：

### 2.1 平台方

1. 电商平台数据控制者
2. 持有订单、支付、收货、地址、用户画像、站内行为、营销触达、售后等数据
3. 负责定义哪些字段能被谁以什么方式使用

### 2.2 商家

1. 在平台上经营店铺
2. 可以看到与自己业务直接相关的数据
3. 不能看到超出履约、售后、经营分析所需的用户敏感信息

典型不能直接给商家的信息：

1. 完整家庭地址的长期留存访问
2. 不属于该商家订单的用户信息
3. 用户跨店行为轨迹
4. 平台级标签和跨商家画像

### 2.3 广告主 / 广告平台

1. 希望与平台做人群重合分析、转化归因、营销效果评估
2. 有自己的用户标识集合
3. 不应直接拿到平台用户明文，也不应把自己的明文用户库交给平台

### 2.4 物流 / 履约方

1. 只应在履约任务范围内获得必要收件信息
2. 不应获得广告、经营分析、平台画像等无关数据
3. 商家本身也不应因为履约而永久掌握完整家庭信息

### 2.5 审计 / 合规 / 策略管理员

1. 定义谁可以发起什么查询
2. 定义哪些字段可用于 join、过滤、聚合、发布
3. 审核最终可发布结果

## 3. 电商场景里的数据分类

建议把数据分成 4 层：

### 3.1 直接标识符

1. 手机号
2. 邮箱
3. 身份证号
4. 收货人姓名
5. 详细家庭地址
6. 设备号

这些字段默认不应被明文扩散。

### 3.2 业务 join 键

1. 规范化手机号
2. 规范化邮箱
3. 平台用户 ID
4. 广告侧 user identifier

这些字段可以在严格边界下参与：

1. SSE 候选筛选
2. bridge token 化
3. PJC 安全求交

但不应作为普通查询结果直接返回。

### 3.3 业务属性

1. 订单金额
2. 商品类目
3. 下单时间
4. 店铺 ID
5. 活动标签
6. 触达渠道
7. 是否复购

### 3.4 高敏感扩展属性

1. 家庭住址
2. 常住商圈
3. 婴幼儿/老人等家庭结构推断
4. 高价值消费标签
5. 健康、宗教、政治等不应被使用的敏感推断

这些字段通常不应出现在商家查询、广告协作、普通分析结果中。

## 4. 典型业务情景

## 4.1 情景 A：平台内部细粒度查询

目标：

1. 平台运营人员要分析“某活动下单但未复购”的用户群
2. 但不希望把全量明文订单表暴露给查询方

推荐路径：

1. 先通过 `SSE` 检索活动关键词、标签或候选记录集
2. 再从受控数据源中仅恢复必要候选记录
3. 只导出允许字段，例如：
   - join key 的受控表示
   - 订单金额
   - 活动标签
   - 是否复购
4. 不导出家庭地址、完整手机号等字段

这个场景里，`SSE` 的价值是：

1. 让平台先从“大量敏感记录”缩小到“少量候选记录”
2. 减少不必要的明文恢复与处理

## 4.2 情景 B：商家经营分析

目标：

1. 商家希望看自己店铺某活动下单用户的复购率、客单价、分层统计

允许结果：

1. 聚合后的购买人数
2. 复购人数
3. 客单价区间统计
4. 类目偏好分布

禁止结果：

1. 用户完整家庭地址明文导出
2. 平台跨店铺标签明细
3. 其他商家的用户行为
4. 可逆推出个人身份的明细结果

这个场景里，平台应当：

1. 先用 `SSE` 找到商家被授权范围内的候选订单
2. 再通过 policy 控制只导出允许字段
3. 如果结果面向商家展示，通常还要经过 release policy 限制

## 4.3 情景 C：平台与广告主做人群交集分析

目标：

1. 广告主有一批待投放用户
2. 平台想知道其中有多少人是已购买用户、复购用户、高价值用户
3. 双方都不想暴露自己的全量用户明文

这就是当前项目里 `PJC` 最匹配的核心场景。

允许结果示例：

1. 交集人数
2. 交集上的订单金额总和
3. 交集上的分桶统计
4. 满足阈值后的归因报告

禁止结果示例：

1. 平台把匹配到的用户手机号列表给广告主
2. 广告主把自己的用户明文名单给平台
3. 返回过细结果导致单个用户被重新识别

## 4.4 情景 D：履约协同

目标：

1. 物流方只在派送任务发生时获取必要收件信息

这个场景和 `PJC` 不是最强相关，和“按职责最小暴露”更相关。

平台应遵守：

1. 商家不因为广告、分析、营销获得家庭住址
2. 物流方不因为履约获得营销标签
3. 地址信息只进入履约链路，不进入商家分析链路

这个场景更强调：

1. 数据分域
2. 字段级策略
3. 服务边界

## 5. 当前项目里 SSE 和 PJC 是怎么协同工作的

当前主链路可以概括为：

```text
SSE 查询/受控导出
-> record recovery 恢复候选记录
-> bridge 标准化与 token 化
-> PJC 安全求交与聚合
-> policy release 治理发布
```

更细一点：

### 5.1 SSE 阶段做什么

`sse/` 当前负责：

1. 存储或管理可搜索加密数据
2. 基于关键词或条件找到候选记录
3. 控制哪些记录、哪些字段允许进入后续流程
4. 生成受审计的 bridge-ready handoff

在电商场景里，SSE 更像：

1. 一个“高敏数据前门过滤器”
2. 一个“先缩小候选集，再进入隐私计算”的阶段

### 5.2 为什么不是 SSE 直接做完全部事情

因为 `SSE` 擅长的是：

1. 搜索
2. 候选集筛选
3. 本平台内部的受控导出

它不擅长的是：

1. 双方都不暴露明文的跨机构 join
2. 广告侧与平台侧的人群交集安全计算

这部分才交给 `PJC`。

### 5.3 bridge 为什么必须存在

`bridge/` 是中间隔离层，不是多余的一层。

它负责：

1. join key 标准化
2. 生成 scoped token
3. 生成 `server.csv` / `client.csv`
4. 写入 bridge metadata 和审计

在电商场景里，它的作用是：

1. 不让 `PJC` 直接接触原始 join key
2. 不让上游直接把各种脏格式标识符送进计算
3. 把“这个 job 用什么 scope、什么密钥、什么 normalizer”固定下来

### 5.4 PJC 阶段做什么

`a-psi/` 当前负责：

1. 对 `server.csv` 和 `client.csv` 做 Private Join and Compute
2. 求交集
3. 在交集上做允许的聚合
4. 产出结果和审计记录

在广告协作场景中：

1. 平台侧输入可以是“购买用户 + value”
2. 广告侧输入可以是“广告用户集合”
3. 最终只得到交集统计，而不是双方明文名单

### 5.5 policy release 最后一层做什么

`policy_release.py` 不是简单“打印结果”。

它负责：

1. 阈值控制
2. 重复查询拒绝
3. 结果治理
4. 生成可公开/可下发报告

这一步是防止：

1. 交集太小导致单用户泄漏
2. 反复试探查询把隐私慢慢抠出来
3. 未经治理的底层结果直接流向业务方

## 6. 当前实现下的任务边界

## 6.1 `sse/` 的边界

负责：

1. 搜索加密
2. 候选筛选
3. 受控导出
4. 加密 record store 恢复
5. recovery service
6. export audit

不负责：

1. 跨机构安全 join
2. PJC 结果治理
3. 最终 public report

## 6.2 `bridge/` 的边界

负责：

1. join key normalizer
2. token 生成
3. `server/client` PJC 输入构造
4. bridge metadata
5. bridge audit

不负责：

1. SSE 搜索
2. 底层 PJC 执行
3. 结果发布策略

## 6.3 `a-psi/` 的边界

负责：

1. PJC 执行
2. 交集聚合
3. policy release
4. duplicate query guard
5. public report

不负责：

1. 明文原始数据筛选
2. join key 标准化
3. 上游 record recovery

## 7. 用电商广告协作来串起来看

下面给一个最匹配当前项目的完整例子。

### 7.1 输入

平台侧持有：

1. 订单数据
2. 用户站内标签
3. 活动标签
4. 用户邮箱/手机号等 join 标识

广告主侧持有：

1. 待投放用户集合
2. 广告转化归因集合
3. 自己的 join 标识

### 7.2 处理

第一步，平台先在 `SSE` 中筛选：

1. 找到“某活动触达过的人”
2. 或“最近 30 天下单用户”
3. 或“某商家授权范围内的用户”

第二步，平台通过 recovery/export 只导出：

1. join key
2. 聚合 value
3. 最小必要属性

第三步，`bridge` 做：

1. join key 规范化
2. job-scope token 化
3. 生成 PJC 输入文件

第四步，`PJC` 做：

1. 与广告主侧 token 化集合求交
2. 在交集上计算订单金额总和、转化人数等

第五步，`policy release` 做：

1. 阈值判断
2. 重复查询防护
3. 只输出允许的聚合结果

### 7.3 输出

广告主最终拿到的应该是：

1. 命中人数
2. 转化人数
3. 聚合金额
4. 分桶统计

广告主不应该拿到：

1. 用户手机号列表
2. 用户家庭地址
3. 平台完整订单明细
4. 单用户级交集明细

## 8. 这个场景下应该补充的能力

如果你要把项目明确定位成“电商隐私库平台”，还建议后续补：

1. 数据域划分：
   - 用户域
   - 订单域
   - 履约域
   - 营销域
2. 字段分级：
   - 明文禁止
   - 仅内部可见
   - 仅 join 可用
   - 仅聚合输出
3. 角色模板：
   - 平台运营
   - 商家
   - 广告主
   - 物流方
   - 审计员
4. 场景模板：
   - 商家经营分析
   - 广告协作归因
   - 平台内部细粒度运营分析
   - 履约最小必要访问

## 9. 建议你在答辩或介绍时怎么讲

建议用下面这句话概括：

1. `SSE` 解决“在平台内部先把需要触碰的数据缩到最小”
2. `PJC` 解决“跨机构协作时只暴露允许的交集统计结果”
3. `policy release` 解决“即使算出了结果，也不一定允许直接发布”

如果再压缩成一句：

1. `SSE` 管入口，`PJC` 管协作计算，`release policy` 管出口。
