# Observability Plan — Seccomp Privacy Platform

## 1. Scope

This plan covers observability for the SSE → Bridge → A-PSI → Policy Release pipeline. All telemetry is derived from **existing audit JSONL and pipeline outputs** — no module internals are modified for metrics collection.

## 2. Telemetry Fields

The following unified fields span all pipeline stages (`config/telemetry_fields.yaml`):

| Field | Type | Source | Notes |
|-------|------|--------|-------|
| `job_id` | string | all audit records | Primary job identifier |
| `correlation_id` | string | all audit records | Cross-stage correlation |
| `caller` | string | SSE export audit, policy audit | Submitting caller identity |
| `tenant_id` | string | derived from policy config | Tenant namespace |
| `dataset_id` | string | SSE export audit source | Dataset identifier |
| `service_id` | string | SSE export audit | SSE service name when applicable |
| `stage` | enum | derived from audit event | sse_export, record_recovery, bridge, pjc, policy_release, audit_seal |
| `status` | enum | audit decision | allow, deny, error |
| `duration_ms` | integer | derived from timestamps | Stage wall-clock duration |
| `row_count` | integer | audit output_rows | Rows processed at this stage |
| `artifact_sha256` | string | audit output_sha256 | Stage output integrity hash |
| `intersection_size` | integer | PJC result | Privacy-safe aggregate |
| `intersection_sum` | integer | PJC result | Privacy-safe aggregate |
| `released` | boolean | public report | Whether the result was released |

### What is NOT recorded

- Raw join keys (email, phone, device ID)
- Token secrets or key material
- Raw record IDs
- Recovery passphrases
- Full record contents

## 3. Metrics

Metrics are generated by `scripts/telemetry_pipeline.py` from existing audit outputs:

| Metric | Type | Description |
|--------|------|-------------|
| `seccomp_job_total` | Counter | Total jobs processed |
| `seccomp_stage_duration_ms` | Histogram | Per-stage duration |
| `seccomp_stage_row_count` | Histogram | Row counts per stage |
| `seccomp_stage_deny_total` | Counter | Deny decisions per stage/reason |
| `seccomp_sse_candidate_count` | Histogram | SSE candidate set sizes |
| `seccomp_pjc_intersection_size` | Histogram | PJC intersection cardinality |
| `seccomp_pjc_intersection_sum` | Histogram | PJC intersection sum |
| `seccomp_release_decision_total` | Counter | Release decisions |
| `seccomp_recovery_boundary_total` | Counter | Recovery boundary type usage |
| `seccomp_artifact_hash_mismatch_total` | Counter | Hash mismatches in audit chain |

## 4. Export Formats

### Phase 1: Text-based export (current)

```bash
# Generate OTLP JSONL metrics from pipeline run
python3 scripts/telemetry_pipeline.py \
  --out-base tmp/sse_bridge_pipeline_demo \
  --tenant-id my_tenant \
  --format otlp-jsonl

# Generate Prometheus text format
python3 scripts/telemetry_pipeline.py \
  --out-base tmp/sse_bridge_pipeline_demo \
  --format prometheus
```

Outputs:
- `<out-base>/telemetry_metrics.jsonl` — OTLP-compatible JSONL metrics
- `<out-base>/telemetry_manifest.json` — collection metadata

### Phase 2: OpenTelemetry Collector (planned)

```
Pipeline outputs → telemetry_pipeline.py → OTLP JSONL file → Otel Collector file receiver → Grafana / Prometheus
```

Integration plan:
1. Configure OpenTelemetry Collector with `filelog` receiver pointing at `telemetry_metrics.jsonl`
2. Add `prometheusexporter` or `otlphttp` exporter
3. Import pre-built dashboard JSON into Grafana

## 5. Alerts

Defined in `config/telemetry_fields.yaml`:

| Alert | Severity | Condition |
|-------|----------|-----------|
| HighStageDenyRate | warning | Deny rate > 10% over 5m |
| PJCIntersectionBelowK | info | Intersection size below k-threshold |
| StageDurationHigh | warning | Any stage > 60s |
| ArtifactHashMismatch | critical | Hash mismatch in audit chain |
| NoRecentJobs | warning | No jobs in 10m |

## 6. Dashboard Layout

Recommended Grafana dashboard rows:

1. **Job Throughput**: job count stat, stage duration heatmap
2. **Stage Health**: deny rate timeseries, recovery boundary usage pie chart
3. **PJC Results**: intersection size histogram, release decision timeseries
4. **Integrity**: artifact hash mismatch counter

Dashboard JSON source is in `config/telemetry_fields.yaml`.

## 7. Runbook Generation

```bash
python3 scripts/generate_observability_runbook.py \
  --out-base tmp/sse_bridge_pipeline_demo \
  --format markdown
```

Produces `docs/runbooks/<job_id>.md` with stage-by-stage health report, latency breakdown, deny decisions, and alert status.

## 8. Phase 2: Production Observability

When moving beyond prototype:

1. Replace `telemetry_pipeline.py` file-based collection with in-process OTLP export from each module
2. Deploy OpenTelemetry Collector as sidecar or daemonset
3. Configure Grafana with the dashboard layout from `config/telemetry_fields.yaml`
4. Add structured logging with `correlation_id` propagation
5. Add distributed tracing across SSE → Bridge → PJC → Release stages
6. Wire alerts into PagerDuty/Slack via Grafana Alerting

## 9. Privacy Constraints

All telemetry must respect these constraints:

- Never log raw email, phone, device ID, or internal user ID
- Never log token secrets, recovery passphrases, or key material
- Never log full record contents
- `canonical_query_signature` may be logged (it is a hash, not a plaintext query)
- PJC intersection aggregates may be logged (they are k-thresholded before release)
- Audit log hashes and row counts are safe to log
